Gantt.IO - Simplified Project Plan

Gantt.IO is a single-page, browser-based project planning tool that stores all data in plain JSON files. Following a file-first philosophy, the project file is the source of truth—readable, editable, and diffable without special tooling. The application runs entirely client-side with no server required, and its architecture allows easy migration to a database backend by swapping only the storage adapter. It provides a Gantt chart with task hierarchy (tasks, groups, milestones), simple finish-to-start and start-to-start dependencies, and integrated work log tracking.

The design uses a dark theme with three background layers (#0f1117 primary, #161922 secondary, #1d2130 tertiary) separated by borders (#2a2f42, #373d54). Text hierarchy uses #d4d8e8 for primary content, #6b7494 for muted secondary text, and #3e4560 for dim tertiary elements. An accent palette of amber (#e8a23a), green (#3dba74), blue (#5b9cf6), purple (#9f7ef5), coral, slate, and orange—each with dim variants—provides color coding for tasks, while red (#e85b5b) indicates errors or warnings. Typography combines IBM Plex Sans for all UI text and IBM Plex Mono for data, labels, and technical elements, with sizes ranging from 9px (row numbers, captions) to 15px (drawer title).

The page structure is a vertical stack: a 44px header containing the logo, project name, team avatars, and action buttons; a 40px toolbar with view switcher (Gantt/List/Board), filter/sort/group controls, zoom controls, search box, and baseline/critical path toggles; a flexible main area split between a resizable sidebar (default 560px, minimum 320px, maximum 60vw) and the canvas; and a 24px status bar displaying counts and overall progress. The sidebar renders a task tree in 32px-high rows across six columns: 26px toggle, 26px type badge (G/group, T/task, P/prototype, M/milestone), title with indentation for hierarchy, percent complete with mini-bar, expected hours, and people pips. Groups have a darker background (#1e2235 on hover), and selected rows use a blue-dim background (#1a2d52).

Interactive behaviors include: a 4px resize handle between sidebar and canvas; collapsible groups with 8x8px chevron toggles; single-row selection with blue-dim highlight; synchronized vertical scroll between sidebar and canvas; sticky task labels that adhere to the left viewport edge when bars scroll off-screen, truncating with ellipsis if space is insufficient; and a slide-up drawer (300px tall) from the bottom for task editing, featuring a drag pill, type badge, title input, field columns, and action buttons. Buttons are consistently 28px tall with primary (amber background) and secondary (bordered) variants, while form inputs use 26-28px heights with subtle borders and focus states.

File Structure (10 files)

  • index.html - HTML structure, no logic
  • style.css
  • app.js - Main orchestrator: global state, file menu, event routing, module coordination
  • core.js - Types (JSDoc), validation, palette, defaults
  • storage/interface.js - Storage API contract (loadProject, saveProject, loadWorklog, saveWorklog)
  • storage/json.js - JSON implementation of storage interface
  • engine.js - Business logic: task tree, dependencies (finish-to-start, start-to-start), worklog aggregation, snapshot rebuilding
  • ui/canvas.js - Gantt canvas rendering: timeline header, task bars, dependency arrows, hit testing
  • ui/sidebar.js - Task list: tree rendering, row selection, scroll sync with canvas
  • ui/drawer.js - Edit drawer: form binding, validation, save/cancel/delete
  • ui/worklog.js - Work log UI: quick entry, history view, aggregation display

File Concerns

app.js

  • Owns global state object (project, worklog, selectedTask, zoom, scroll, etc.)
  • Handles file menu: new, open, save, save-as
  • Routes events between modules (e.g., task selected → update drawer + canvas)
  • Manages view mode (Gantt only, List not implemented per requirements)
  • Coordinates snapshot rebuild after worklog changes
  • No business logic, no rendering, no direct DOM manipulation

core.js

  • Defines all types via JSDoc comments
  • Validator functions: validatePerson, validateTask, validateDependency, validateWorkEntry, validateProject
  • Default factories: defaultPerson, defaultTask, defaultDependency, defaultWorkEntry
  • Palette definition and accessor functions (getColorStop, getColorStops)
  • No dependencies on other modules

storage/interface.js

  • Defines abstract storage API:
    • loadProject(file) → Project
    • saveProject(project, file)
    • loadWorklog(file) → WorkEntry[]
    • saveWorklog(entries, file)
  • No implementation, only function signatures and JSDoc

storage/json.js

  • Implements storage/interface.js using JSON and File System Access API
  • Handles file picker fallbacks
  • Serializes/deserializes Project and WorkEntry[] to/from JSON
  • No business logic, only I/O

engine.js

  • Task tree operations: getChildren, getDescendants, getSiblings, getPath, getRootTasks
  • Rollup operations: rollupGroup (computes group hours, dates, completion from children)
  • Dependency resolution:
    • computeEarliestStart(taskId, tasks, dependencies) → Date
    • getDependencyShift(task, tasks, dependencies) → number (days)
    • Cycle detection
  • Worklog aggregation:
    • aggregateByTask(entries) → Map<task_id, {hours_spent, last_entry_date}>
    • rebuildSnapshots(project, worklogEntries) → Project (updates hours_spent, completion_derived)
  • Color resolution: resolveColor(task, tasks, projectMeta) → PaletteKey
  • No rendering, no file I/O

ui/canvas.js

  • Renders Gantt chart on HTML Canvas element
  • Timeline header: day/week/month/quarter columns based on zoom level
  • Task bar rendering:
    • Background track (full planned duration, pastel color)
    • Progress fill (width = completion_active, mid color)
    • 100% completion = full color
    • Milestones = orange diamond
    • Groups = bracket line
  • Dependency arrows (elbow-style, neutral color)
  • Hit testing: hitTest(x, y) → task_id | null
  • Scroll/zoom synchronization with sidebar
  • No business logic, reads pre-computed data from engine

ui/sidebar.js

  • Renders task list as a tree with indent levels
  • Column cells: type badge, title, % complete mini-bar, expected hours, people pips
  • Row selection state (single select)
  • Collapse/expand for groups
  • Scroll synchronization with canvas
  • Emits events: onRowSelect, onRowCollapse
  • No edit logic, no business logic
  • Clicking an item will open a drawer. to edit the task
  • it is implemented as a virtualized list. each task is a same height div to make virtualization easier.

ui/drawer.js

  • Slide-up bottom panel for editing selected task
  • Form fields: title, type, parent, dates, estimated hours, priority, people, notes, manual completion
  • Dependency editor: add/remove dependencies with rule selector
  • Validation feedback (inline)
  • Actions: save (calls engine), delete (with confirmation), discard
  • No file I/O, no rendering beyond the form

ui/worklog.js

  • Quick-entry dialog: date, hours, person, task, note
  • Work history view: table of entries, inline editable, deletable
  • Summary panel: hours per person, hours per task
  • Actions: addEntry (calls engine), deleteEntry
  • No aggregation logic (lives in engine), no file I/O (lives in storage)
